🔍 Контекст

«Арвато Рус» — независимый сервис-провайдер в сфере электронной коммерции и управления цепочками поставок. Компания ежемесячно обрабатывает до 1 миллиона отправлений, работая с такими службами доставки, как DPD, СДЭК, Boxberry, Почта России, PickPoint и другими.

Я работала в компании «Текмэн» на позиции системного аналитика. Мне была поставлена задача разработать техническое задание на проектирование TMS (Transportation Management System) — системы управления транспортом — с нуля.

❗ Проблема

До создания TMS процессы в компании были ручными и разрозненными:

  • Данные об отправлениях хранились в двух несвязанных ERP-системах (SAP и FullHouse), которые не имели общей базы данных и не были интегрированы между собой.
  • Отчётность по статусам доставки и тарифам служб доставки вносилась вручную сотрудниками.
  • Каждая служба доставки использовала собственную OMS с нестандартизированными данными и статусами — это делало сравнение и аналитику невозможными.
  • Сводная отчётность по разным службам доставки формировалась вручную путём сведения данных из SAP и FullHouse.

Итог: высокая трудоёмкость, ошибки, запаздывающая аналитика и отсутствие единого инструмента для принятия решений по работе со службами доставки.

🎯 Задача

Разработать техническое задание для системы TMS, которая:

  1. Автоматизирует взаимодействие между всеми участниками цепочки доставки: WMS, OMS служб доставки, заказчиками и клиентами.
  2. Унифицирует статусы доставки от разных служб в единый формат.
  3. Обеспечит аналитику и отчётность в режиме реального времени.
  4. Предоставит разграниченный доступ для разных ролей пользователей: сотрудников «Арвато Рус», внешних заказчиков и гостей.

⚙️ Что я сделала

1. Интервьюирование заказчика и погружение в предметную область

Провела серию интервью с представителями разных подразделений «Арвато Рус»: отдела по работе со службами доставки, бухгалтерии, менеджеров по работе с заказчиками, IT-директора. Изучила текущие процессы взаимодействия между SAP/FullHouse, OMS служб доставки и внутренними командами.

2. Описание процессов и требований

Описала полный жизненный цикл отправления: от импорта заказа из интернет-магазина в WMS — через формирование и передачу отправления в службу доставки — до получения статусов, обработки возвратов и формирования итоговой отчётности для заказчиков.

Разработала глоссарий системы (TMS, WMS, OMS, ERP, SAP, FullHouse, статусы, роли) для единого понимания терминов командой разработки и заказчиком.

3. Проектирование архитектуры и ролевой модели

Спроектировала архитектуру системы из трёх ключевых компонентов:

  • АРМ Администратора — управление пользователями, справочниками, отчётами.
  • Функциональные механизмы — обработка тарифов, аналитика, трекинг отправлений.
  • База данных — централизованное хранилище с 9 ключевыми таблицами (Users, Delivery_services, Delivery_Rates, Parcel и др.).

Описала ролевую модель с 7 ролями пользователей (Администратор, Сотрудник отдела доставки, Менеджер по работе с заказчиками, Сотрудник сервисной службы, Сотрудник бухгалтерии, Заказчик, Гость) и разграничением прав доступа для каждой.

4. Описание API-интеграций

Описала структуру данных, передаваемых по API между WMS и TMS (50+ полей заказа) и между TMS и OMS служб доставки. Разработала перечень запросов к API 6 служб доставки: Boxberry, PickPoint, DPD, СДЭК, СПСР, МаксиПост.

5. Унификация статусов служб доставки

Разработала механизм перевода разнородных статусов 10 служб доставки в единые унифицированные статусы TMS «Арвато Рус». Учла краевые случаи: добавление новых статусов, уведомления администратора, дополнительные расшифровки по статусу «Не доставлено» для каждой службы отдельно.

6. Проектирование аналитической подсистемы

Спроектировала 9 аналитических отчётов с описанием фильтров, логики расчётов и структуры данных:

  • Отчёт по срокам доставки
  • Детализация выполнения контрольных сроков
  • Отчёт по возвратам
  • Отчёт о тарифах служб доставки
  • Отчёт по выкупленным отправлениям
  • Отчёт по срокам перечисления денежных средств
  • Сводный отчёт по условиям служб доставки

7. Создание интерактивных прототипов и описание UI

Совместно с командой создала интерактивные макеты в Axure для всех экранных форм системы. Написала подробное описание каждого элемента интерфейса: полей ввода, таблиц, фильтров, кнопок, сообщений об ошибках и всплывающих окон.

Описала все сообщения об ошибках системы с HTTP-кодами и текстами для пользователей.

8. Требования к документации и приёмо-сдаточным испытаниям

Описала структуру пользовательской документации (руководство пользователя, руководство администратора, описание API) и программу приёмо-сдаточных испытаний.

📊 Результат

  • Разработано полное техническое задание объёмом более 130 страниц, охватывающее архитектуру, ролевую модель, API-интеграции, аналитическую подсистему и пользовательский интерфейс.
  • ТЗ стало основой для разработки системы, заменившей ручные процессы обработки данных от 10 служб доставки.
  • Унифицированная статусная модель позволила сравнивать и анализировать данные по всем перевозчикам в единой системе.
  • Разграниченная ролевая модель обеспечила доступ к данным как внутренним сотрудникам (6 ролей), так и внешним заказчикам.
